# v1.0

## Findings

### lacon explain non-functional — raw output never captured on opt-in  {#lacon-explain-non-functional-raw}

<!-- fields -->
- disposition: fixed
- evidence: Audit root cause: run.rs:275 hard-coded raw=None and RunOutcome carried only byte counts, so no live lacon run ever populated raw_outputs (seam 6 broken at capture; headline E2E flow broken). Closed in Phase 7: RunOptions.capture_raw + RunOutcome.raw_captured wired in crates/lacon-core/src/runtime/mod.rs; run.rs sets capture_raw from cfg.store_raw_outputs and passes raw_captured to tracker.record; an E2E test now drives lacon run -> explain rather than a seeded INSERT.
- severity: blocking

### REQ-cli-explain partial — reader implemented, capturer absent  {#req-cli-explain-partial-reader}

<!-- fields -->
- disposition: fixed
- evidence: explain.rs read/replay side (open_readonly -> fetch_invocation -> fetch_raw_output -> Runner::filter_bytes -> two-column diff) was correct and tested, but every real lacon explain hit the 'no stored raw output' branch. Same root cause as the capture gap; closed in Phase 7 so lacon explain is functional on real opt-in data.
- severity: blocking

### REQ-tracking-raw-outputs-default-off — opt-in half non-functional  {#req-tracking-raw-outputs-default}

<!-- fields -->
- disposition: fixed
- evidence: Off-by-default half was satisfied; the opt-in half fired the privacy warning (record.rs:62) but stored nothing because run.rs passed raw=None. Fixed in Phase 7 — flipping store_raw_outputs:true now populates raw_outputs per docs/specs/tracking-data-model.md.
- severity: blocking
