# 0014: `lacon stats` read-time presentation layer

**Status:** Accepted (2026-05-23)

## Context

`lacon stats` reads the four reporting views defined in ADR 0011 / `docs/specs/tracking-data-model.md` (`v_unmatched_offenders`, `v_filtered_offenders`, `v_bypass_rate`, `v_project_savings`) and prints each one in full, one row per line, in the order the view returns them. The command was specified for the common case: a handful of projects, a handful of rules.

Real usage on the maintainer's machine exposed three problems that make the output effectively unreadable:

1. **The per-project list grows without bound.** `v_project_savings` groups by `project_path`, which is the literal `current_dir()` captured at invocation. Tools that run inside throwaway temp directories (test harnesses, `cat /tmp/...`) produce a unique `project_path` per run, so the "Per-project savings" section degrades into hundreds of one-run rows. (A separate cause — integration tests that did not isolate `XDG_DATA_HOME` writing into the developer's real `history.db` — was fixed in `crates/lacon-cli/tests/{end_to_end,cli_run}.rs` and is not in scope for this ADR.)

2. **Git worktrees fragment one project into many.** A worktree at `<repo>/.claude/worktrees/agent-XXX` (or anywhere else — worktree placement is tool-defined) records its own `project_path`, so a single repo worked on across N worktrees appears as N unrelated projects. The same fragmentation happens for runs launched from a repo *subdirectory*.

3. **The columns are jargon and partly mislabeled.** "Offenders" is author shorthand. `filtered_bytes` is the count of bytes that *survived* filtering and reached the model, but the name reads as "bytes removed." `keep_ratio` is `kept / raw`, where **lower is better** — counterintuitive next to a byte column, and inverted relative to the "saved" framing used in the project section. Byte counts are printed as raw integers. No section is capped, and there is no overall summary line.

Constraints that bound the fix:

- **Append-only migrations** (ADR 0011): the four views live in migration `0001` and must not be edited; a view change requires a new migration.
- **Cold-start budget ≤ 10 ms on the hook hot path** (ADR 0001, tightened by ADR 0013): `lacon run` records on every matched command, so nothing may be added to the *write* path that resolves filesystem state per invocation.
- **`command_normalized` is implementation-defined** (`tracking-data-model.md`) and already groups cosmetically-different invocations; project identity has no equivalent.
- **`cli_stats.rs` snapshot tests** (D-11) pin the current plain-text output and will be regenerated by any format change.

Canonicalizing a `project_path` to a stable "which repo is this" key needs filesystem access (reading `.git`), which cannot run inside SQL and must not run on the write hot path.

## Decision

Add a **read-time presentation layer** to `lacon stats`. The stored data model is unchanged: the write path still records the literal logical `current_dir()` as `project_path`, the four views are untouched, and no migration is introduced. All of the following happens in `lacon-cli` (`commands/stats.rs` plus a small helper module), with one new aggregate reader behind the `lacon-core::tracking::query` boundary (D-01).

1. **Overall headline.** A summary line is printed first: total runs, distinct projects (after canonicalization), `raw → kept` bytes, and `saved` (absolute + percent), over `bypassed = 0` rows. Backed by a new `query::overall_totals(conn)` aggregate (and a `--since`/`--project`-filtered counterpart).

2. **Project canonicalization + rollup.** The "Savings by project" section reads the existing per-`project_path` rows and re-aggregates them in Rust under a *canonical key*. Re-aggregation is exact because every project-savings field is an additive sum (runs, raw, filtered, saved). The canonical key of a path is computed, in order:

   a. **Ephemeral.** If the path is under a temporary root (`std::env::temp_dir()`, plus the platform temp prefixes `/tmp`, `/var/folders`, `/private/var/folders`, and `$TMPDIR`), the key is the single synthetic bucket `(ephemeral)`. All temp-dir runs collapse into one line.

   b. **Repo root via read-time `.git` resolution.** Otherwise, walk up the path's ancestors looking for `.git`:
      - `.git` is a **directory** → repo root is that ancestor (handles normal repos and runs launched from a subdirectory).
      - `.git` is a **file** (`gitdir: <p>`) → the path is a *linked worktree*. Resolve `<p>`, read `<p>/commondir`, resolve it to the main `.git` directory, and take its parent as the repo root. This maps any tool's worktree — wherever placed — back to its main repository.

      No `git` subprocess is spawned; this is a bounded sequence of file reads, and it runs only inside `stats` (a cold path), never on the write hot path.

   c. **Literal fallback.** If no `.git` is found (including when the recorded directory no longer exists on disk), the key is the literal `project_path`, unchanged. Behavior never regresses below the pre-change exact path.

   Ephemeral detection takes precedence over `.git` resolution, so a throwaway repo created under a temp root still collapses into `(ephemeral)`.

3. **Top-N capping.** Every section prints at most N rows (default 10), ordered by its primary metric, followed by a `… M more` line with a drill-in hint (`--project` / `--rule` / `--since`, or a future `--all`). This bounds output regardless of history size.

4. **Clarified columns and labels.** Section headers and columns are reworded for a reader who did not write the schema: "offenders" framing is replaced with task-oriented labels (e.g. "Commands with no rule", "Rule effectiveness"); the surviving-bytes column is named `sent`/`kept` rather than `filtered_bytes`; effectiveness is shown as `saved %` (higher is better) to match the project section instead of the inverted `keep_ratio`. Byte counts are humanized (`22.8 KB`), with a `--bytes` flag to print exact integers for scripting.

The stored field names (`filtered_bytes`, `avg_keep_ratio`) and the view definitions are **not** renamed — the change is confined to the CLI presentation. `tracking-data-model.md` is unaffected; the relabeling is documented wherever the `stats` output is described.

## Consequences

- **No schema migration, write path untouched.** The hot-path cold-start budget (ADR 0001 / 0013) is unaffected; canonicalization cost is paid only in `stats`.
- **Project grouping reflects current filesystem state, not invocation-time state.** A repo that has since moved or been deleted falls back to its literal recorded path. This is acceptable for a savings report and is the price of keeping resolution off the write path.
- **Rollup is exact for project savings** (sums only). It deliberately does **not** attempt to merge the `AVG`-based `keep_ratio` of `v_filtered_offenders`, which groups by command+rule and is unaffected by project canonicalization.
- **`cli_stats.rs` snapshots are regenerated** to the new format; the empty-DB and exit-code contracts (D-03, exit 2 on bad `--since`) are preserved.
- **`stats` gains a dependency on filesystem layout** (reading `.git`). It remains read-only and opens the DB via `open_readonly` (D-02); the new logic is pure path/file handling with a literal-path fallback on any I/O error.
- **Generality over the Claude-Code case.** Resolution is via the git worktree mechanism, not the Claude-specific `.claude/worktrees/<id>` path, so worktrees created by any tool, in any location, roll up correctly.
- **Reversibility.** If project identity later needs to survive deleted directories or to appear in SQL grouping, a `repo_root` column can be added by a future append-only migration computed on the write path; this ADR does not foreclose that, it defers it to avoid hot-path cost.

## Relationship to prior ADRs

Additive. No existing ADR is amended. It builds on ADR 0011 (SQLite tracking / the four views) and respects the cold-start budget from ADRs 0001 and 0013. The `tracking-data-model.md` schema and views are unchanged.
