<objective>
Build the tracking READ surface inside `lacon-core` that `stats` and `explain` (Wave 2) consume. Phase 2 shipped a write-only tracking layer; this plan adds the read path: a non-mutating read-only DB open helper (D-02), a `tracking::query` module (D-01) exposing typed per-view rows + base-table filtered re-queries (D-09) + explain's row/BLOB lookups (D-05), and the Wave-0 spike that decides whether strict read-only open works on a WAL `history.db` (Open Question 1 / Pitfall 1).

Purpose: All SQL stays behind the `lacon-core` boundary (D-01, architecture.md); `lacon-cli` keeps `rusqlite` dev-only. This is the data layer Wave 2 commands plug into.
Output: `tracking::query` module, `open_readonly` helper on the tracking module, the Wave-0 WAL read-only spike, and `tests/tracking_query.rs`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filter args (`--project`/`--rule`/`since`) → SQL | Untrusted CLI strings reach query parameters (resolved in Wave 2, but the query API is the binding point) |
| history.db file → read-only Connection | A user-writable SQLite file is opened; a read command must never mutate it |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | `tracking::query` SQL with `--project`/`--rule`/`since` | mitigate | All read queries use `?N` placeholders + `params![]`; filter values are NEVER string-interpolated into SQL (Task 3 acceptance gate; record.rs precedent) |
| T-04-02 | Tampering | `open_readonly` against a writable history.db | mitigate | `SQLITE_OPEN_READ_ONLY` (or fallback without CREATE); no migrate/prune/INSERT; Task 2 + Task 4 prove COUNT unchanged after open |
| T-04-03 | Information disclosure | reading another user's DB | accept | DB lives under the invoking user's XDG data dir with `0700` dir perms (Phase 2); no cross-user path is opened — read path resolves only `xdg_db_path()` |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No new packages this phase (RESEARCH Package Legitimacy Audit: zero installs); nothing to verify |
</threat_model>
