<objective>
Close the single root-cause gap from the v1.0 milestone audit: capture the pre-filter (raw) bytes of a `lacon run` invocation when `store_raw_outputs` is enabled and persist them to the existing `raw_outputs` table, so `lacon explain <id>` reproduces a REAL invocation end-to-end instead of only hand-seeded SQL rows.

The reader side (`explain.rs`, `Runner::filter_bytes`, `query::fetch_raw_output`), the write API (`Tracker::record(meta, raw_opt, …)`, `insert_raw_output`, `RawOutput`), and the privacy double-gate are ALL already built and tested. Only the capture path is missing: `run.rs:275` hard-codes `None`, so opting in produces an empty `raw_outputs` table and every real `lacon explain` hits the "no stored raw output" branch.

Purpose: Make `lacon explain` functional on real data, satisfying REQ-acceptance-explain-reproducibility (byte-exact reproduction) and the opt-in half of REQ-tracking-raw-outputs-default-off / REQ-cli-explain.
Output: A gated capture field on `RunOutcome`, a `RunOptions` flag, the `run.rs` wiring that constructs `RawOutput` and passes `Some(&raw)`, plus a true E2E byte-exact test and a `RunOutcome`-shape unit test.

Scope discipline (from CONTEXT.md): the remediation is "modest, not architectural." OUT of scope — do NOT build: redaction of `raw_outputs`, `lacon purge`, encryption-at-rest, separate real stderr capture, any new schema/columns, a seventh CLI command. No ADR is amended.
</objective>
