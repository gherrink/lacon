<design_notes>
## Why `validate_rule()` cannot just call `RuleLoader::resolve()` for the standalone path

`validate_file(path)` is called by both:
1. The `lacon validate <path>` CLI (where the file is whatever the user passed — could be anywhere on disk).
2. The library-level `validate_dispatch.rs` integration tests (same).

`RuleLoader::resolve(rule_id)` walks the layered `<project>/.lacon/rules/`, `~/.config/lacon/rules/`, and bundled directories looking up by `id`. That is the WRONG path for "validate this specific file at this path" — the file may not be in any of those directories.

## Three valid wiring strategies (pick exactly one)

**Strategy A (lightest — recommended): direct call to `compile_pipeline` for stages, plus a `flatten_extends_with_lookup` call to detect cycles using a same-directory parent-lookup closure.**

- After `parse_one()` succeeds, run `flatten_extends_with_lookup(rule, path, &mut HashSet::new(), &|parent_id, child_path| { find_rule_in_dir(parent_id, child_path.parent().unwrap_or(Path::new(".")), child_path) })`.
- That gives back a flat `RuleFile` (or `CircularExtends` error) — `flatten_extends_with_lookup` already detects cycles via `visited: &mut HashSet<String>`.
- Then call `compile_pipeline(flat.pipeline.unwrap_or_default(), path, 32768)` to compile regexes (catches `InvalidRegex`).
- Then if `flat.on_error` is `Some`, call `compile_pipeline(on_err.pipeline, path, 32768)` for the on_error pipeline too.
- Then if `flat.post_process` is `Some`, call `resolve_script_path(&pp.path, path)` (already in `loader.rs`, but it's `fn` not `pub fn`) — OR call `compile_resolved(flat, path, RuleSource::Project, 32768)` which already calls `resolve_script` internally.

**Strategy B (heaviest — accepts and discards the StarlarkScript): call `compile_resolved(flat, path, RuleSource::Project, 32768)` directly after flatten.**

- This compiles regexes AND parses Starlark scripts (which is more than strictly needed for validate — but it's the pre-built API, and parsing Starlark catches malformed scripts at validate time, which is a nice bonus).
- The `RuleSource::Project` argument is a synthetic value because the file may not actually be in the project layer; for validate, the source is just metadata that's unused after compile.

**Strategy C (use `find_rule_in_dir` for parent lookup but call `compile_resolved`): combine A's flatten with B's compile.**

**RECOMMENDED: Strategy B.** Reasons:
1. Single call site — minimum new code.
2. `compile_resolved` already does `flatten` internally? NO — it does NOT. Re-read `loader.rs:484-526`: `compile_resolved` does NOT flatten extends. The caller (i.e. `parse_flatten_compile`) flattens first and then passes the flat rule. So validate_rule must do the flatten step itself, then call compile_resolved.

So the actual recommended sequence:
```
parse_one(content, path)          → RuleFile (raw, possibly with extends)
flatten_extends_with_lookup(...)  → flat RuleFile or CircularExtends
compile_resolved(flat, ...)       → ResolvedRule or InvalidRegex/MissingScriptFile/ParseError
```

Discard the resulting `ResolvedRule` — we only care about whether compile succeeded.

## Parent lookup for `extends`

For `lacon validate <path>`, the parent rule lookup should walk the parent's directory (typically `.lacon/rules/` or wherever the file is). The cycle-detection test fixtures are `cycle_a.yaml` and `cycle_b.yaml` — both must be in the same directory for cycle detection to work end-to-end.

**For the CLI test:** copy BOTH `cycle_a.yaml` and `cycle_b.yaml` into a tempdir's `.lacon/rules/` and run `lacon validate cycle_a.yaml`. The validate path must resolve `extends: cycle-b` by reading `cycle_b.yaml` from the same directory.

**Closure to pass to `flatten_extends_with_lookup`:** identical pattern to `RuleLoader::parse_flatten_compile` in `loader.rs:265-268`:
```rust
let dir = path.parent().map(|p| p.to_owned()).unwrap_or_else(|| PathBuf::from("."));
let lookup = |parent_id: &str, child_path: &Path| -> Option<Result<RuleFile, ValidationError>> {
    find_rule_in_dir_pub(parent_id, &dir, child_path)
};
```

But `find_rule_in_dir` is **private** in `loader.rs` (line 334). One of:
- (i) Make `find_rule_in_dir` `pub` in `loader.rs`.
- (ii) Inline the same logic inside `validate/mod.rs`.

**Pick (i)** — a 1-character change (`fn` → `pub fn`) avoids duplicating directory-walk code. Document in the action block.

## Defaults max_bytes constant

Use `32768` matching `RuleLoader::new`'s default. Or: lift a `pub const DEFAULT_MAX_BYTES: usize = 32768` into `loader.rs` and reuse from both. Pick whichever is smaller; both acceptable. Recommendation: define `pub const DEFAULT_MAX_BYTES: usize = 32768;` in `loader.rs` and reference from both `RuleLoader::new` and `validate_rule`.

The current comment at `validate/mod.rs:124-132` explicitly states this is a known shortcut. **Delete the entire comment block** and replace with a comment explaining the new wired-in compile pass.

    ```rust
    /// Validate a rule file at `path` with content `content`.
    ///
    /// Wires the full compile pipeline so SC4 (Phase 1 success criterion) is satisfied:
    /// 1. Schema parse via `parse_one` — catches `UnknownKey` (deny_unknown_fields) and
    ///    `ParseError` (malformed YAML, including unknown StageSpec variants like `reverse_lines`).
    /// 2. `extends` flattening via `flatten_extends_with_lookup` with a same-directory parent
    ///    lookup closure — catches `CircularExtends`. Parents are looked up in `path.parent()`.
    /// 3. `compile_resolved` — catches `InvalidRegex` (regex compile fails) and
    ///    `MissingScriptFile` (Starlark script path absolute / contains `..` / does not exist).
    ///
    /// The resulting `ResolvedRule` is discarded — we only care whether the compile pass succeeded.
    fn validate_rule(path: &Path, content: &str) -> Vec<ValidationError> {
        // Step 1: schema parse.
        let rule = match parse_one(content, path) {
            Ok(r)  => r,
            Err(e) => return vec![e],
        };

        // Step 2: flatten extends. Look up parents in the same directory as `path`.
        // For files outside any rules directory (ad-hoc validation), this still works
        // because `find_rule_in_dir` returns `None` when the parent ID is not present —
        // which produces a `ParseError` ("could not find parent rule") for the user.
        let parent_dir = path
            .parent()
            .map(|p| p.to_owned())
            .unwrap_or_else(|| std::path::PathBuf::from("."));

        let mut visited = std::collections::HashSet::new();
        let lookup = |parent_id: &str, child_path: &Path| {
            crate::rules::loader::find_rule_in_dir(parent_id, &parent_dir, child_path)
        };

        let flat = match crate::rules::loader::flatten_extends_with_lookup(
            rule,
            path,
            &mut visited,
            &lookup,
        ) {
            Ok(f)  => f,
            Err(e) => return vec![e],
        };

        // Step 3: compile pass. Catches InvalidRegex, MissingScriptFile, and any
        // remaining ParseError surfaces (e.g. Starlark parse failure).
        match crate::rules::loader::compile_resolved(
            flat,
            path,
            crate::rules::loader::RuleSource::Project, // synthetic — unused after compile
            crate::rules::loader::DEFAULT_MAX_BYTES,
        ) {
            Ok(_resolved) => Vec::new(),
            Err(e)        => vec![e],
        }
    }
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user-supplied rule file → `lacon validate` parser | The validate command accepts arbitrary YAML files at arbitrary paths. The file content is untrusted (user may type any YAML); the file path is also untrusted (user may pass `/tmp/foo.yaml` or `../../etc/foo.yaml`). |
| rule file's `script.path` field → filesystem read | A rule's `script:` references a path on disk. After this plan, validate now resolves that path. Same trust boundary as `RuleLoader::resolve` already crosses (PLAN-03 mitigation). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-validate-01 | Tampering / Information disclosure | `validate_rule` script-path resolution via `compile_resolved` → `resolve_script` → `resolve_script_path` (loader.rs:695-739) | mitigate | Reuse PLAN-03's existing T-03-04 path-traversal guard. `resolve_script_path` rejects absolute paths and any `Component::ParentDir` (`..`) — this fires inside `compile_resolved`, so `lacon validate /tmp/evil.yaml` where `evil.yaml` says `script: { path: /etc/passwd }` produces `MissingScriptFile` and exit 1. No code path lets the attacker read `/etc/passwd` via validate. **Verification:** add a CLI test in a future hardening plan that asserts `lacon validate /tmp/path_traversal.yaml` does NOT open `/etc/passwd` (out of scope for this gap-closure plan; the unit-level guarantee is already covered by `path_traversal_rejected` in `rules_loader.rs:163-182`). |
| T-validate-02 | Denial of service (ReDoS) | `compile_pipeline` → `compile_regex` for user-supplied patterns | accept | The `regex` crate (D-03 dependency choice) provides linear-time matching by construction — no catastrophic backtracking is possible regardless of input. Compilation itself is bounded by `regex` crate's parser; pathological patterns return `InvalidRegex` rather than spinning. ReDoS on the validate path is structurally impossible with `regex` (not `fancy-regex`). **No additional mitigation needed.** |
| T-validate-03 | Information disclosure (error messages) | `ValidationError` Display includes the file path | accept | The path printed is the path the user supplied — they already know it. Stderr is the user's own terminal in the CLI case; no cross-user disclosure surface in v1 (no daemon, no shared service). |
| T-validate-04 | Tampering (extends parent lookup outside expected dir) | `find_rule_in_dir` walks `path.parent()` for parent-rule lookups | accept | The walk is bounded to `read_dir(parent_dir)` — a single non-recursive directory listing. An attacker who controls a rule file in dir X can cause validate to read other `.yaml` files in X to find the parent, but only YAML files in X — no symlink chasing, no escape from X. Dir X is wherever the user pointed `lacon validate`. Attack surface is identical to "user passes a rule file from a directory containing other YAML files" — already the normal case for `.lacon/rules/`. |
