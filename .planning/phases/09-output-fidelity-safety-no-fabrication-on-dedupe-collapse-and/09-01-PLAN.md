<objective>
Fix the inline `LACON_DISABLE=1 <cmd>` env-prefix bypass in the Claude Code PreToolUse hook (REQ-adapter-bypass-detection). Today `detect_bypass` only reads the hook process's own environment, so an inline assignment on the command string never bypasses — the highest-leverage bug because `!!` is unusable from inside the agent's Bash tool, making `LACON_DISABLE` the agent's only escape hatch.

Per D-01..D-04: add a leading-`NAME=value` scan to `detect_bypass` that short-circuits to `PassThrough` BEFORE any chain split or wrap (D-02). Reuse the existing exact-`"1"` value semantics. No engine change — `run_bypassed` is already the byte-exact backstop (D-05), which this plan also locks with a regression test (success-criterion #2, engine half).

Purpose: Restore the user's trust guarantee that bypass always works on the hot path.
Output: Extended `detect_bypass` + unit tests (quoting variants, leading-position-only, negative) + a hook-e2e inline-prefix passthrough test + an engine-level byte-exact `run_bypassed` assertion.
</objective>

| Hook → real shell | On PassThrough the unmodified command string reaches the shell, which applies the leading assignment and runs the command unwrapped. |

| T-09-01 | Tampering | inline-env-prefix parser in `detect_bypass` | mitigate | Bypass is the SAFE direction (filtering off = original bytes pass through, never fabricated). The real threat is MIS-detection that drops filtering on a NON-bypass command. Mitigated by leading-position-only scan (break at first non-assignment token) + exact-`"1"` value match after one-layer unquote — `echo LACON_DISABLE=1` and `LACON_DISABLE=0` provably do not bypass (negative unit tests T D-04). |
| T-09-02 | Denial of Service | inline-env-prefix parser (hot path, ≤10ms budget) | mitigate | Parser is an O(prefix-length), allocation-light `split_whitespace` leading scan that breaks at the command word; no full POSIX grammar (Don't Hand-Roll). Cold-start budget preserved (RESEARCH A3). |
