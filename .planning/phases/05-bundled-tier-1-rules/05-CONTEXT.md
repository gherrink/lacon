# Phase 5: Bundled Tier 1 rules - Context

**Gathered:** 2026-05-22 (assumptions mode)
**Status:** Ready for planning

<domain>
## Phase Boundary

Ship ten Tier 1 YAML rules in `bundled-rules/` (`pkg-install`, `cargo-build`, `cargo-test`, `vitest`, `jest`, `pytest`, `tsc`, `eslint`, `git-status`, `docker-build`), each with a success-path and a failure-path fixture under `tests/fixtures/<rule-id>/<scenario>/` (`input.txt`, `expected.txt`, `meta.yaml`), plus a single fixture-based integration test (`cargo test --test bundled_rules`) that asserts ≥50% byte reduction on primary success fixtures and zero error-line drops — all hermetically (CI never installs pnpm/cargo/vitest/etc).

**In scope:** authoring the 10 rule YAMLs against the existing engine, capturing fixtures, and building the one walking test runner. **Out of scope:** any new engine primitive, new CLI command, the v1 ship-gate acceptance/docs (Phase 6), and Tier 2 rules. The engine, loader, `extends`, and `Runner::filter_bytes` are all already shipped (Phases 1–4) — Phase 5 only *consumes* them.

Depends on: Phase 1 (engine + primitives + loader + `filter_bytes`) only. Strictly, Phase 5 needs nothing from Phases 2–4 except `Runner::filter_bytes` (Phase 4).
</domain>

<decisions>
## Implementation Decisions

### A. Test-runner mechanism

- **D-01:** The `bundled_rules` integration test runner is **subprocess-free byte replay**. For each fixture it: loads the rule by id via `RuleLoader::new(None).resolve("<rule-id>")` (project_dir `None` → resolves straight from the bundled embedded layer, hermetic), constructs `Runner::new(resolved, RunOptions::default())`, and runs `runner.filter_bytes(&input_bytes, exit_code, 0, "<command>", None)`. It never spawns a process. This mirrors `lacon explain` (`crates/lacon-cli/src/commands/explain.rs:118-159`), the closest working analog.
- **D-02:** Each fixture's `meta.yaml` carries a new **`exit_code` field** (success fixtures `0`, failure fixtures nonzero). The runner passes this to `filter_bytes`, which selects the ADR-0010 branch (`exit_code == 0` → success pipeline; `!= 0` with `on_error` present → `on_error` pipeline; else raw passthrough). **The `meta.yaml` shape documented in `docs/specs/testing-rules.md` does NOT include `exit_code` — adding it is a required deliverable**, and the doc's `meta.yaml` schema must be updated to match. Without it, failure fixtures would silently run through the success pipeline and never exercise `on_error`.

### B. Fixture authoring

- **D-03:** `input.txt` is **real captured tool output** (merged stdout+stderr), lightly trimmable/anonymizable but not hand-synthesized — per the "real captures are the contract" stance in `docs/specs/testing-rules.md`. `expected.txt` is generated by running the rule's own pipeline against `input.txt` (the regeneration recipe), not authored by hand. CI installs none of the ten tools; fixtures are static checked-in text.
- **D-04:** Byte-exact comparison follows the **established `tests/primitives.rs:42` idiom**: join the filtered lines with `\n` and compare against `expected.trim_end_matches('\n')` (tolerates a single editor-added trailing newline). The runner must replicate this normalization or the whole suite goes red on cosmetic trailing newlines.
- **D-05:** Per-fixture assertions, all driven from `meta.yaml`: (1) byte-exact match against `expected.txt`; (2) reduction threshold `len(expected)/len(input) <= 0.5` on primary success fixtures, skippable via `exempt_from_reduction_check: true` (failure fixtures that are already small are exempt); (3) opt-in `must_keep_lines` substring list — every listed substring must survive filtering (the explicit "the error must survive" encoding).

### C. Rule pipeline design & `extends`

- **D-06:** The four same-shape test-runner rules — `cargo-test`, `vitest`, `jest`, `pytest` — **share a base via `extends`** (`extends: bundled/cargo-test`, or a dedicated `bundled/test-base`), each overriding only its `match`. Cross-bundled `extends` is implemented (`loader.rs:584-620`, `merge_rules` prepends parent pipeline + inherits scalar fields) but has **no existing fixture-level test** — Phase 5's test runner will be the first to exercise it. **Fallback if it proves buggy at resolve time:** copy the shared pipeline into each of the four rules (the spec-sanctioned "copy the parent" escape). The planner should treat verifying the bundled→bundled extends path as an early/spike task.
- **D-07:** Rule authors **never hand-place `max_bytes`** — the loader auto-injects `Stage::MaxBytes` (`DEFAULT_MAX_BYTES = 32_768`) onto both the success and `on_error` pipelines when absent (`loader.rs:674-695`). Only add an explicit `max_bytes` to override the default cap. `keep_regex` stages OR-merge into a `RegexSet`. `script:` is **rejected inside `pipeline:`** at load time — Starlark is available only via the top-level `post_process` block (ADR-0008).
- **D-08:** Per-rule reduction strategy (whitelist `keep_regex` vs blacklist `drop_regex` vs `collapse_repeated`/`keep_around_match`) is **Claude's discretion at planning time**, guided by `docs/bundled-rules-roadmap.md` per-rule notes and the captured real output. Failure-path filtering uses `on_error` (context-preserving: keep errors + tail) per ADR-0010; success-path filtering is the reductive pipeline.

### D. Test target location & assertion style

- **D-09:** The test lives at **`crates/lacon-core/tests/bundled_rules.rs`** (cargo auto-discovers `tests/*.rs`; the filename is what makes `cargo test --test bundled_rules` resolve). Fixtures are reached with the established workspace-root idiom `env!("CARGO_MANIFEST_DIR")` joined with `../../tests/fixtures/<rule-id>/<scenario>/`. Assertions use **plain `assert_eq!`** (the entire existing suite does; `insta` is a declared dev-dep with zero usages — do not introduce it).

### E. Match scope for package-manager script aliases

- **D-10:** Bundled rules match **concrete tool invocations** only (`vitest`, `vitest run`, `jest`, `npx jest`, `eslint`, `tsc`, `tsc --noEmit`, `cargo test`, `cargo build`/`cargo check`, `pytest`/`python -m pytest`, `git status`, `docker build`/`docker buildx build`). The resolution-dependent aliases `pnpm test` / `pnpm lint` (which run jest *or* vitest *or* eslint depending on the project) are **NOT matched by bundled rules in v1** — `lacon` cannot know what they resolve to. Users wanting them filtered add a project rule. `pkg-install` directly matches `npm install` / `pnpm install` / `yarn install` / `pnpm i` / `pnpm add`.
- **D-11:** `pkg-install` MAY carry a `rewrite` block (e.g. `add_flags: ['--reporter=silent']`) — but the exact quiet/silent flag per package manager (npm vs pnpm vs yarn) must be confirmed by research before it is added; an unsupported flag would break the wrapped command. If a single flag isn't universally safe, prefer pipeline-side filtering over a risky rewrite.

### Claude's Discretion

- Whether the shared test base is `bundled/cargo-test` itself or a separate `bundled/test-base` rule (D-06) — pick whichever reads cleaner; if a separate base, it still must be a loadable rule with a valid `match` (or be excluded from resolution, planner's call).
- Exact regex patterns and stage ordering inside each rule's success pipeline (D-08).
- Plan decomposition / batching of the 10 rules (e.g. by family: pkg-install; cargo-build; the 4 test runners; tsc+eslint; git-status; docker-build) vs one-rule-per-plan — planner's call, optimizing for the wave-based parallel executor.
- Scenario slugs beyond the required success+failure pair (e.g. `clean-install`, `with-warnings`, `compile-error`).

<specifics>
## Specific Ideas

- The ten rule ids are fixed by REQ-bundled-rules-tier1: `pkg-install`, `cargo-build`, `cargo-test`, `vitest`, `jest`, `pytest`, `tsc`, `eslint`, `git-status`, `docker-build`.
- Per-rule reduction intent (from `docs/bundled-rules-roadmap.md`): cargo-build drops "Compiling foo v0.x" repeats, preserves warnings/errors with file:line; the 4 test runners preserve the summary line + failures, drop per-test PASS lines; pytest drops dot-progress, preserves tracebacks; tsc is mostly ANSI-strip + dedupe + tail (the output IS the signal); eslint drops "passing" summaries; git-status collapses huge untracked-files sections; docker-build drops layer cache hits, preserves build steps + errors.
</specifics>
