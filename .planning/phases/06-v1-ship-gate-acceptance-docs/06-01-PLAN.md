<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| stored raw build log → `lacon explain` rendering | Bytes captured from an untrusted subprocess are replayed; the filtered "safe view" column is shown to a human in a terminal. |
| test process → developer real home dir | New tests could write to `~/.claude/settings.json` / `~/.local/share/lacon/history.db` if not sandboxed. |
| external tool (pnpm) → CI lane | The real-pnpm path performs a network `pnpm install` (supply-chain surface) and must never enter the hermetic CI lane. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | `lacon explain` filtered column (terminal-control injection via stored raw build logs) | mitigate | The D-03 byte-equality test asserts the FILTERED column re-derivation; it must NOT regress the existing C0/C1/ESC neutralization in `explain.rs` (WR-01). The RAW column stays verbatim by design (SC3); the safe-view neutralization stays intact. Test reads only seeded bytes. |
| T-06-02 | Tampering | new tests writing to real user state | mitigate | Every new/modified test redirects `XDG_DATA_HOME`/`XDG_CONFIG_HOME` to a tempdir and uses a tempdir cwd (Pitfall 4). `git status` shows no `.claude/settings.json` mutation after a test run. |
| T-06-03 | Tampering (supply chain) | real `pnpm install` in `pnpm_e2e_real` | mitigate | The real-pnpm test is `#[ignore]`d so default `cargo test` and CI never invoke it; the hermetic variant uses the in-repo `test_emitter` stub (no network, no registry). Tracked in Plan 02's CI as T-06-SC. |
| T-06-04 | Spoofing | locating the `test_emitter` / `lacon` binaries | mitigate | Resolve via `assert_cmd::cargo::cargo_bin(...)` (cargo artifact), never a PATH lookup that a planted binary could hijack (T-07-04 precedent). |

No high-severity (block-on-high) threats are introduced by this plan: it adds tests and a planning doc only, runs hermetically, and the one network-touching path is `#[ignore]`d out of CI. The supply-chain concern of the real `pnpm install` is real but contained to an opt-in, manually-run test.
</threat_model>
