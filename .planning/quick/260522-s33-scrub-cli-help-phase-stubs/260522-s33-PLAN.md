# Quick Task 260522-s33: Scrub stale phase-stub annotations from CLI help strings

## Problem

User-facing `lacon --help` output leaked development-phase annotations that were
never removed after the commands were implemented. Each was a doc comment on a
`CliCommand` variant in `crates/lacon-cli/src/cli.rs`, which clap renders verbatim
into both top-level and per-subcommand help:

```
init      Set up `.lacon/`, install Claude Code hook, add CLAUDE.md note. (Phase 3 implementation — Phase 1 stub.)
stats     Show top offenders, bypass rates, unmatched commands. (Phase 4 implementation — Phase 1 stub.)
explain   Re-run filtering against stored raw output for a tracked invocation. (Phase 4 implementation — Phase 1 stub.)
doctor    Verify hooks installed, configs valid, rules parse. (Phase 4 implementation — Phase 1 stub.)
```

All four commands are fully implemented in v1.0, so the "(Phase X implementation —
Phase 1 stub.)" lines are stale and misleading to users.

## Scope

- IN: the four `/// (Phase X implementation — Phase 1 stub.)` doc-comment lines on
  `Init`, `Stats`, `Explain`, `Doctor` in `crates/lacon-cli/src/cli.rs`.
- OUT: internal `// Phase N` developer comments elsewhere in the codebase — these
  document historical evolution and cross-phase contracts, are not user-facing, and
  were not part of the report.

## Task

1. Delete the four stale doc-comment lines in `cli.rs`.
   - verify: `cargo build -p lacon-cli && ./target/debug/lacon --help` shows no
     "Phase" / "stub" text; `grep` finds no remaining stale strings.
   - done: help output is clean and `cargo test -p lacon-cli` passes.
